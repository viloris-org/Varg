# Aster Language Family Specification

## Status

Draft. This document records the current direction for Aster's scripting, declaration, and interchange languages.

## Motivation

Aster should not expose one universal script language for every kind of game authoring task. AI and human creators need to declare scenes, models, audio, lighting, physics, behavior, and runtime logic. These concerns have different safety, validation, readability, reuse, and execution requirements.

The scripting system is therefore split into a small language family:

- declarative languages for assets, scenes, and behavior;
- one graph/runtime-capable language for logic that actually needs computation.

The main design constraint is that AI should modify structured declarative files whenever possible, and only use a Turing-complete script when the task requires dynamic logic. A second constraint is that not every Aster language has the same audience: some are AI-first, some are shared by AI and humans, and some are implementation or interchange formats.

## Language audience classes

Every Aster language must declare its audience class before its syntax is designed.

| Audience class | Primary author | Human readability | Compatibility burden | Example burden | Validator burden |
| --- | --- | --- | --- | --- | --- |
| AI-only interchange | AI/tools | Low to medium | Low | High for model output examples | Very high |
| AI-first human-readable | AI, reviewed by humans | High | Medium | Very high | Very high |
| Shared authoring | Humans and AI | Very high | High | High | High |
| Human-first AI-assisted | Humans, edited by AI | Very high | Very high | Medium | High |
| Engine-internal IR | Engine/tools | Low | Internal only | Low | Very high |

Definitions:

- **AI-only interchange** is optimized for generation, parsing, and validation. Humans may inspect it, but it is not the primary editing surface.
- **AI-first human-readable** is optimized for AI generation while still being pleasant enough for humans to review and patch.
- **Shared authoring** is a real user-facing language. Humans and AI both write it, so syntax stability, readable diagnostics, and documentation matter more.
- **Human-first AI-assisted** is a conventional scripting surface where AI helps, but humans must be able to own long-term maintenance.
- **Engine-internal IR** is not an authoring language. It must be stable for tools, but it should not be marketed as user syntax.

The audience class determines the specification style. Do not write every language spec the same way.

## Specification profiles

Each language should have one of these documentation profiles.

### AI-only interchange profile

Use for machine-facing formats such as canonical JSON operations, generated behavior JSON, context packets, or transaction bundles.

Required spec sections:

- purpose and non-goals;
- schema;
- canonical serialization;
- trust label and authority rules;
- allowed and forbidden fields;
- validation errors;
- migration/versioning;
- golden examples;
- fuzz or schema tests.

Human tutorial content is optional. Validator behavior is mandatory.

### AI-first human-readable profile

Use for `.amdl`, `.ascn`, and `.abv` unless a language graduates into shared authoring.

Required spec sections:

- purpose and file role boundary;
- allowed responsibilities;
- disallowed responsibilities;
- grammar or syntax summary;
- short examples by task;
- common mistakes and diagnostics;
- validation rules;
- compilation target;
- reference rules;
- formatter expectations;
- migration/versioning;
- model prompt guidance.

The language must have many small examples because models do not know it from pretraining.

### Shared authoring profile

Use when a language is meant to be edited directly by both humans and AI.

Required spec sections:

- all AI-first human-readable sections;
- human tutorial;
- style guide;
- refactoring rules;
- editor affordances;
- stable backwards compatibility policy;
- error recovery behavior;
- examples for both "written by human" and "generated by AI" workflows.

Shared authoring languages must avoid syntax that is easy for AI but unpleasant for humans, and must avoid clever human syntax that models frequently misgenerate.

### Human-first AI-assisted profile

Use for runtime scripting surfaces like `.as` once humans are expected to maintain complex logic.

Required spec sections:

- execution model;
- capability model;
- standard library;
- lifecycle hooks;
- type system or dynamic semantics;
- sandbox and side-effect rules;
- diagnostics;
- debugging story;
- examples and anti-examples;
- AI generation constraints;
- compatibility policy.

The spec must distinguish "AI may generate this" from "AI should prefer this." For `.as`, AI should prefer declarative `.abv`, `.ascn`, or `.amdl` when possible.

### Engine-internal IR profile

Use for compiled representations that users should not hand edit.

Required spec sections:

- producer and consumer modules;
- versioning;
- invariants;
- validation;
- serialization if persisted;
- compatibility with old artifacts;
- tests.

No prompt examples are required unless models produce the IR directly.

## Core language tiers

| Tier | Language | Extension | Audience class | Purpose | Turing complete |
| --- | --- | --- | --- | --- | --- |
| Model declaration | Aster Model | `.amdl` | AI-first human-readable | Reusable model, mesh, material, collider, mass, LOD, and asset composition declarations. | No |
| Scene declaration | Aster Scene | `.ascn` | AI-first human-readable | Scene graph, entities, transforms, light, audio, physics environment, initial state, and references to models, prefabs, behaviors, and scripts. | No |
| Behavior declaration | Aster Behavior | `.abv` | AI-first human-readable | Behavior trees, state machines, selectors, sequences, conditions, and high-level AI actions. | No by default |
| Runtime logic | Aster Script | `.as` | Human-first AI-assisted | Event handlers, gameplay rules, dynamic systems, procedural logic, quest rules, UI logic, and other runtime computation. | Yes |

The MVP language set is:

```text
.amdl  Aster Model
.ascn  Aster Scene
.abv   Aster Behavior
.as    Aster Script
```

## Reserved future extensions

These extensions are reserved but should not be implemented until the corresponding domain becomes complex enough to justify a separate language.

| Language | Extension | Purpose |
| --- | --- | --- |
| Aster Prefab | `.apfb` | Reusable entity or entity-group declarations. |
| Aster Material | `.amat` | Standalone material and shader-parameter declarations. |
| Aster Audio | `.aud` | Audio events, emitters, mixers, buses, ambience, and spatial audio declarations. |
| Aster World | `.awld` | Multi-scene world layout, level streaming, global rules, and campaign structure. |

## File role boundaries

### `.amdl` — Aster Model

Audience class: AI-first human-readable.

Use `.amdl` for reusable asset composition. A model file describes what an object is, not where it is placed in a scene.

Allowed responsibilities:

- mesh or primitive references;
- material references or inline simple material parameters;
- collider presets;
- mass and physical material defaults;
- attachment points;
- LOD declarations;
- animation set references.

Disallowed responsibilities:

- scene placement;
- gameplay event handlers;
- per-level state;
- arbitrary loops or runtime control flow.

Example:

```astermodel
amdl

model Crate {
  mesh {
    kind = primitive.box
    size = [1, 1, 1]
  }

  material {
    kind = inline
    base_color = "#8a5a2b"
    roughness = 0.75
    metallic = 0.0
  }

  collider {
    shape = box
    size = [1, 1, 1]
  }

  rigidbody {
    mode = dynamic
    mass = 12kg
  }
}
```

### `.ascn` — Aster Scene

Audience class: AI-first human-readable.

Use `.ascn` for world composition. A scene file declares entities, transforms, light, audio, physics environment, and references to reusable assets.

Allowed responsibilities:

- entities and hierarchy;
- transforms;
- model, prefab, behavior, and script references;
- lights;
- audio sources and listeners;
- physics world parameters;
- initial component values.

Disallowed responsibilities:

- procedural generation loops;
- complex gameplay logic;
- long-running behavior logic;
- reusable model definitions that should live in `.amdl`.

Example:

```asterscene
scene ForestArena {
  gravity = [0, -9.81, 0]

  light Sun {
    kind = directional
    direction = [-0.4, -1.0, -0.2]
    intensity = 4.0
  }

  audio AmbientForest {
    clip = "audio/forest_loop.ogg"
    loop = true
    volume = 0.4
  }

  entity Crate01 {
    model = "models/crate.amdl"
    transform = position([3, 0.5, 2])
    rigidbody = dynamic
  }
}
```

### `.abv` — Aster Behavior

Audience class: AI-first human-readable.

Use `.abv` for structured AI and entity behavior. This is the evolution path for the existing declarative behavior-tree work.

Allowed responsibilities:

- selectors, sequences, parallels, decorators, and conditions;
- state-machine transitions;
- high-level engine actions;
- references to reusable presets;
- bounded parameters for patrol, chase, flee, attack, interact, and similar behaviors.

Disallowed responsibilities:

- arbitrary computation as the default path;
- asset declarations;
- scene layout;
- low-level frame-by-frame movement code when a high-level action exists.

Example:

```asterbehavior
behavior EnemyGuard {
  selector {
    sequence {
      condition player.distance < 12
      action chase(player)
      action attack_if_in_range(player)
    }

    action patrol(points: ["A", "B", "C"])
  }
}
```

### `.as` — Aster Script

Audience class: human-first AI-assisted.

Use `.as` only for runtime logic that requires Turing-complete computation.

Allowed responsibilities:

- event handlers;
- gameplay rules;
- custom calculations;
- procedural generation;
- quest and mission rules;
- UI logic;
- integration glue that cannot be represented declaratively.

Disallowed responsibilities:

- bulk scene declaration;
- static model declaration;
- behavior that can be safely represented in `.abv`;
- unbounded authority over engine systems without an explicit capability grant.

Example:

```asterscript
on update(dt) {
  if input.key_down("Space") && player.is_grounded {
    player.velocity.y = 6.5
  }
}
```

## Reference rules

Files should reference lower-level or runtime files by path or stable asset ID:

```text
.ascn -> .amdl
.ascn -> .abv
.ascn -> .as
.abv  -> .as only for explicit custom action hooks
```

Model files should not reference scenes. Scene files may reference models, behaviors, scripts, and future prefabs. Behavior files may call named engine actions and may reference `.as` hooks only when a declarative action is insufficient.

## Compilation model

Each language should compile into a stable intermediate representation before runtime use.

Expected pipeline:

```text
.amdl  -> Asset/model IR -> engine-assets / engine-render / engine-physics
.ascn  -> Scene IR       -> engine-ecs scene file / runtime world
.abv   -> Behavior IR    -> engine-script-declarative behavior tree
.as    -> Runtime IR     -> Rhai today, possible AsterScript VM later
```

JSON remains useful as an interchange or generated artifact format, but it should not be the primary authoring format for AI-facing game creation.

## Compatibility with current implementation

Current repository state already contains:

- `engine-script-declarative` for behavior-tree style scripting;
- JSON behavior examples under `examples/behaviors/`;
- `engine-script-rhai` for runtime-capable scripting;
- JSON scene and prefab examples under `examples/project/`.

This specification does not require deleting those formats. The migration path is:

1. keep existing JSON scene, prefab, and behavior files as supported interchange formats;
2. add parsers for `.amdl`, `.ascn`, and `.abv` that compile to the existing Rust data structures;
3. keep Rhai as the first implementation backend for `.as` until a dedicated Aster Script runtime is justified.

## Existing and transitional formats

Aster may keep existing JSON formats and Rhai support while the language family matures.

| Format | Current role | Audience class | Direction |
| --- | --- | --- | --- |
| Behavior JSON | Existing behavior-tree interchange | AI-only interchange | Keep as tool/interchange format; `.abv` becomes authoring surface. |
| Scene JSON / prefab JSON | Existing scene and prefab interchange | AI-only interchange | Keep as import/export or migration target; `.ascn` and future `.apfb` become authoring surfaces. |
| Rhai scripts | Existing runtime scripting backend | Human-first AI-assisted | Keep as backend for `.as` or legacy compatibility until Aster Script runtime is justified. |
| Agent operation JSON | Model/tool operation protocol | AI-only interchange | Specify by schema and execution gate, not by prose prompt examples alone. |

These formats should not all receive the same kind of documentation. JSON protocols need schemas, canonical examples, and validators. Human-facing languages need syntax guides, style rules, diagnostics, and editor behavior.

## Design rules

- Prefer declarative files for AI-generated assets, scenes, and behavior.
- Keep `.amdl`, `.ascn`, and `.abv` non-Turing-complete by default.
- Keep `.as` Turing-complete, sandboxed, capability-gated, and visibly distinct from declarative files.
- Do not describe every language as AI-only. Declare whether it is AI-only, AI-first, shared, human-first, or internal.
- Do not use `.ast` as the primary extension; it is too generic and does not preserve the language tier.
- Avoid long extensions such as `.astermodel` for normal project authoring; short suffixes are the canonical form.
- Preserve clear ownership: a file should describe one kind of thing.
- Make every declarative language statically checkable before runtime.
- Design syntax for AI generation first: regular structure, explicit names, few hidden defaults, and stable diagnostics.
- For AI-first languages, maintain many short examples and diagnostic repair examples.
- For human-first languages, maintain tutorials, stable semantics, editor/debugging behavior, and compatibility rules.
- For interchange formats, treat schema and validator behavior as the spec authority.
