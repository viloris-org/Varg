# Strong Commander/KeyPool Branch Audit

> Date: 2026-06-26
> Auditor: Claude Code goal-mode session
> Branch: feat/ai-native-engine-loop (副轨 review, not implementation branch)
> Status: Architecture review — not yet a branch, still design/docs phase

## Summary

The Commander/KeyPool pack proposes a hierarchical AI coding architecture where a single top-tier
"Commander" model owns all hard decisions and delegates bounded tasks to lower-tier subagents,
routed through a multi-channel KeyPool with 429 cooldown, health scoring, and fallback chains.
The architecture is designed for 8-hour autonomous coding sessions via a durable execution ledger,
structured compaction, and resume tokens.

**Verdict**: The direction is sound but the documents are written for a different codebase
("SokachCode V6.4") and need significant adaptation before any implementation in Aster.
Key gaps: no Rust types, no async runtime, existing engine-agent-cluster overlaps unaddressed,
stream racing cost model incomplete for Chinese APIs.

---

## Top 5 Problems Found

### P1: SokachCode references do not match Aster

Documents reference "SokachCode V6.4" with a 16-filter router, 39 passing KeyPool tests,
a 14-phase goal state machine, and a 7-tab goal panel. Aster is a Rust game engine.
There is no existing router scoring engine, no KeyPool, no goal state machine.
The implementation plan is written for a different product.

### P2: Language/data model mismatch — all types are TypeScript, Aster is Rust

Every data model in KEYPOOL_ROUTER_SPEC.md is TypeScript. The ChatGPT research uses Python.
No Rust type definitions with serde derive exist. Translating union types, discriminated
unions, and optional fields to Rust enums with proper serde is non-trivial and unaddressed.

### P3: Tournament compaction is under-specified

The `TournamentState`/`Scorecard` compaction replaces conversation history with summaries,
but doesn't specify **what model generates the summaries**, with **what prompt**, within
**what token budget**. The `lastFullState?: string` field creates an unacknowledged memory
explosion problem (serializing the entire GoalState recursively).

### P4: Overlaps with existing engine-agent-cluster

Aster already has `Manager` trait (decompose, request_grants, merge, request_deep_review,
create_repair_ticket, generate_final_report), `Worker` trait, `TaskAssignment`,
`WorkerOutput`, `ReviewDecision`, `RepairTicket`, `ContextPacket`, `CapabilityGrant`.
The proposed Commander/SubagentTask/ExecutionLedger system duplicates this with different
terminology and no migration mapping.

### P5: Stream racing has no cancellation safety for Chinese APIs

GLM (Zhipu), Qwen (DashScope), DeepSeek use SSE streaming. HTTP-level abort does NOT
stop server-side token generation or billing. Racing N slots where all N complete but
only 1 is used has a real cost impact that is acknowledged but not solved.

## Top 5 Improvement Opportunities

### I1: Extend engine-agent-cluster + engine-policy instead of duplicating

Add a `CommanderProfile` struct to `engine-policy::context` specifying model tier and
fallback chain. Extend `Manager::decompose()` to accept a CommanderProfile. This avoids
duplicating the entire protocol layer.

### I2: Define KeyPool as a Rust module in engine-ai with serde types

Create `keypool.rs` in `engine-ai` with `#[derive(Serialize, Deserialize)]` structs for
`KeyPool`, `KeySlot`, `ProviderChannel`, `HealthScore`, `FallbackChain`. Port the
`handle429` function as a method. Write deterministic tests. This is the actual MVP.

### I3: Replace Tournament compaction with PhaseSummary

The ChatGPT research already defines `PhaseSummary` with concrete fields: phase,
duration_ms, key_outcomes, decisions_made, files_created_or_modified, unresolved_issues,
recovery_hint, next_step_ref. Specify the commander model generates these via a
200-token structured output call. Remove Scorecard/loser_pile complexity.

### I4: Embed Router into AgentSession via AiModel trait

Add a `Router` struct implementing `AiModel` that delegates to a selected `KeySlot`.
`AgentSession::plan()` already takes `&dyn AiModel` — the Router becomes the implementor
instead of a bare provider. Minimal-change integration.

### I5: Define Execution Ledger as JSONL append-only file

Both research sources agree on JSONL append-only. Define an `ExecutionLedger` with
`append()` and `query(filters)` methods. Do NOT use an in-memory ring buffer as primary
store — it doesn't survive crashes.

## Missing Pieces for Implementation

1. **No Rust types** — all models are TypeScript/pseudocode
2. **No provider adapters** — no DashScope (Qwen), Zhipu (GLM), DeepSeek, SiliconFlow connectors
3. **No Commander system prompt** — no spec for how the Commander's prompt differs from the
   existing `system_prompt::build()`
4. **No async runtime** — Aster uses blocking I/O for AI calls; proposed architecture needs
   tokio/async-std for concurrent subagents, stream racing, and background timers
5. **No persistence layer** — no file paths, rotation policies, or corruption recovery for
   Ledger, GoalState, or KeyPool state
6. **No frontend integration** — no Tauri commands, IPC protocol, or panel components specified
7. **No test strategy** — references "39 existing KeyPool tests" that don't exist in Aster
